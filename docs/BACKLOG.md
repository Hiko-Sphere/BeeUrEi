## A. 远程视频协助（决定"能不能真用"，整体 P0–P1）

### A1. 真实来电唤醒（PushKit + CallKit + APNs）❌ · P0
- **目标**：协助者/亲友 App 在后台或锁屏时，盲人发起求助能像真实来电一样**响铃并唤起**通话界面；接听后进入 WebRTC 通话。
- **达成标准**：
  - 盲人点"一键求助" → 被叫方手机即使息屏也弹出系统级来电 UI（CallKit），响铃≤3 秒内到达。
  - 接听后双方进入同一 `callId` 房间，音频接通；盲人长按才出画面。
  - 拒接/未接有回退（转下一位紧急联系人）。
  - 通话计入历史记录。
- **技术要点**：iOS `PushKit`(VoIP push) 注册 token → 上报后端；后端用 **APNs**(VoIP 证书) 推送来电 → 客户端收到后用 `CallKit` `CXProvider` 报告来电 → 接听触发信令 join。后端需存"用户↔VoIP token"，紧急路由按优先级逐个推送。
- **依赖**：🌐 Apple Developer 账号（VoIP 推送证书）。

### A2. WebRTC 真机媒体打通 🟡 · P0
- **目标**：把已写好的 `WebRTCMediaEngine`（`#if canImport(WebRTC)` 守卫）在真机上真正跑通音视频。
- **达成标准**：两台真机（一盲一协助）同一 callId，**音频双向、盲人长按出视频、协助者能看到画面**，弱网下不崩；端到端延迟 < 1 秒（同城）。
- **技术要点**：Xcode 加 SPM 包 `https://github.com/stasel/WebRTC`；真机编译激活真实路径；按需修正个别 API；加麦克风/摄像头权限串与用途说明。
- **依赖**：🌐 stasel/WebRTC 包 + 两台真机。

### A3. coturn TURN 服务器实部署 🔜（短时效凭据系统+/api/assist/turn 端点+iOS ICE 接线+coturn 配置已就绪；coturn 实部署待开 UDP 安全组端口）· P0
- **目标**：跨运营商/NAT 的两端也能接通（不只是同 Wi‑Fi）。
- **达成标准**：双方分别在 4G 与家庭宽带下仍能接通；信令把 TURN 凭据下发到客户端 `RTCIceServer`；连通率在测试样本中 > 95%。
- **技术要点**：在 awsjapan 跑 coturn（已存 `coturn.conf.example`）；用短期 TURN 凭据(HMAC)由后端签发；开放 UDP 端口；客户端 ICE servers 加 STUN+TURN。
- **依赖**：服务器端口/安全组（经 Cloudflare 或独立公网端口）。

### A4. 协助者在线状态 / 匹配 / 排队（presence）✅（presence+匹配已上线；排队扩圈待 A1 推送）· P1
- **目标**：盲人求助时，系统知道"谁在线、派给谁"，而不是只返回名单。
- **达成标准**：协助者切"在线待命"后进入可用池；盲人非定向求助时，后端按"在线+语言+空闲+评分"选一位推送，无人接则排队/扩圈；超时回退志愿者池。
- **技术要点**：后端维护 presence（内存→规模化用 Redis）；WS 心跳维持在线；匹配算法（核心可单测：`HelperMatcher` 按在线/语言/负载打分）。
- **依赖**：后端（presence 存储）。

### A5. 通话录制实际落地（可选合规）❌ · P2
- **目标**：把"录制策略"从元数据变成真实录制（用户已同意时）。
- **达成标准**：双方同意后可录制本次通话音/视频；文件**端到端或静态加密**存储；到期按保留期自动删除；用户可在 App 内申请删除。
- **技术要点**：客户端录制（`RTCVideoTrack` → 文件）或服务端 SFU 录制；加密(KMS/信封加密)；与后端 retention 联动。
- **依赖**：存储(对象存储/加密)。

## B. 避障感知（核心体验，P1 为主）

### B1. 动态 ROI 碰撞走廊真机调参 + 地面估计 🟡 · P1
- **目标**：让动态 ROI 在真机上**贴合真实前方走廊**，默认开启替代静态 ROI。
- **达成标准**：真机上绿框稳定贴合"前方与身等宽、地面到头高、前向自适应"的区域；走/转/上下坡时不漂移；开启后障碍召回不降反升；可设为默认。
- **技术要点**：用 `ARPlaneAnchor`(水平面) 估**真实地面高度**替换当前 1.2m 近似；走廊 origin 用地面点；加 IMU 重力对齐；真机标定相机外参与约定；调 W/H/N 默认值。
- **依赖**：真机调参。

### B2. 红绿灯/过街可靠识别 🟡 · P1
- **目标**：路口"红灯请等待/绿灯可通行"达到**可信赖**水平。
- **达成标准**：在常见人行横道信号灯场景，白天判对率 > 90%，**绝不把红判成绿**（安全偏保守）；远距给"前方有信号灯"，足够近才判色；夜间/眩光下显式说"无法确认"。
- **技术要点**：现启发式(YCbCr 采样)仅 MVP；正式需**专用红绿灯检测/分类模型**（行人信号灯数据集微调）；加时间投票稳定；结合人行横道检测。
- **依赖**：🌐 标注数据/模型训练；真机调参。

### B3. 视觉‑惯性世界系稳像接入 ARKit ❌ · P1
- **目标**：把检测从抖动图像帧投影到**重力对齐世界系**测方位，从源头抗手抖（规格 §3 已设计，未接运行时）。
- **达成标准**：手持轻抖时方位/距离读数稳定（方差较现状下降明显）；转头不漂移；`limited` 跟踪时冻结更新而非跳变。
- **技术要点**：用 `SensorFrame.camera`(已加) 把检测框 + LiDAR 深度反投影到世界系算 bearing/range（核心 `PinholeCamera` 已测）；喂入 `ObstacleTracker`。
- **依赖**：真机验证。

### B4. 高危地形检测：台阶/下坡/坑洞/落差 ✅（核心+接入已上线，真机调参）· P1
- **目标**：识别"向下台阶、马路牙子、坑、落差"这类**最危险且 COCO 模型识别不了**的地形。
- **达成标准**：接近向下落差(≥15cm)时提前预警"前方下台阶/落差"；上行台阶提示；准确率与误报率在真机达到可接受阈值。
- **技术要点**：纯 LiDAR 深度几何分析——地面平面拟合 + 前方深度突变/法向变化检测（可下沉核心 `GroundHazardDetector` 单测）；不依赖分类模型。
- **依赖**：真机调参。

### B5. 检测模型正式训练（中文/高危类别）🌐 · P1
- **目标**：从 demo COCO 升级到**面向街景高危类别**的模型（台阶/路桩/玻璃门/消火栓/电线杆/共享单车等）。
- **达成标准**：目标类别在自采街景测试集 mAP 达标；端侧推理 ≥ 10fps；中文标签直出。
- **技术要点**：采集+标注街景数据 → 微调 YOLO → Core ML 导出(含 NMS)；我可写训练/转换/评测脚本。
- **依赖**：🌐 你提供/采集标注数据。

### B6. 真机性能实测（发热/帧率/延迟/功耗）🟡 · P0（上架前）
- **目标**：在目标机型矩阵上确认**可持续运行**且安全降级。
- **达成标准**：连续避障 30 分钟不过热停机；端到端"看到→播报"延迟 < 400ms；功耗可接受；`ThermalPolicy`/`PowerPolicy` 降级阈值经实测校准。
- **技术要点**：Instruments/真机埋点测帧率、热状态、CPU/Neural Engine 占用、电池；校准降级参数。
- **依赖**：真机。

## C. 导航（P1–P2）

### C1. 国内高德 iOS SDK 实时逐向 🌐 · P1
- **目标**：国内从"步骤读出 MVP"升级到**实时逐向 + GCJ‑02 定位跟随**。
- **达成标准**：国内定位连续跟随、按距离触发转向播报、偏航重规划，与海外 MapKit 体验一致。
- **技术要点**：集成高德/百度**iOS 导航 SDK**（测绘法+GCJ‑02 纠偏，纯端侧不可行）；复用核心 `RouteProgress`/`OffRouteDetector`。
- **依赖**：🌐 高德/百度 SDK + key + 资质。

### C2. 室内/公交地铁/动态码/"最后十米" ❌ · P2
- **目标**：补齐竞品已有但 BeeUrEi 缺的导航维度。
- **达成标准**：支持室内逐向(GoodMaps 式)或 NaviLens 动态码识别引导；公交/地铁换乘提示；到达后"最后十米"用相机+OCR/招牌识别引导找门。
- **技术要点**：评估 NaviLens SDK / 室内地图 / 相机 OCR；分子项立项。
- **依赖**：🌐 第三方 SDK/数据。

### C3. 导航真机定位实测 🟡 · P1
- **目标**：海外 MapKit 链路真机验证可用。
- **达成标准**：真实步行场景下信标方向正确、转向播报时机合适、偏航能重规划、AirPods 头追踪生效。
- **依赖**：真机 + 室外。

## D. 账号 / 后端 / 基础设施（P1–P2）

### D1. 账号安全完善 🔜（refresh token 轮换/撤销已上线；邮箱/手机验证、找回密码待做）· P1
- **目标**：refresh token 轮换、邮箱/手机验证、找回密码。
- **达成标准**：access token 短期 + refresh 轮换（可吊销）；注册需验证邮箱/手机；忘记密码可重置；防撞库（限速+锁定）。
- **技术要点**：后端加 refresh 表 + 轮换/吊销；验证码服务（邮件/短信网关）；bcrypt 已有。
- **依赖**：🌐 邮件/短信服务商。

### D2. 数据持久化升级 + 备份 ❌ · P1
- **目标**：从单容器 SQLite 升级到可备份、可扩展的存储。
- **达成标准**：迁移到 **Postgres**（Store 接口已隔离，低成本切换）；每日自动备份 + 可恢复演练；presence/队列用 **Redis**。
- **技术要点**：实现 `PostgresStore`；备份脚本 + 异地存储；docker‑compose 编排。
- **依赖**：服务器资源。

### D3. 监控 / 告警 / 日志 ❌ · P1
- **目标**：线上可观测、出问题能告警。
- **达成标准**：健康探针 + 关键指标(在线数/通话成功率/错误率) 面板；异常告警到你的渠道；结构化日志聚合可检索。
- **技术要点**：Prometheus/Grafana 或托管方案；Fastify 日志接入；告警 webhook。
- **依赖**：监控栈。

### D4. 后端水平扩展 ❌ · P2
- **目标**：支撑规模化并发与高可用。
- **达成标准**：多实例无状态 + Redis 共享 presence/信令路由；负载均衡；通话信令在多实例间正确路由。
- **依赖**：D2 完成。

### D5. 基础设施加固 / 自动化 ❌ · P1
- **目标**：生产级运维。
- **达成标准**：源站 TLS（或确认隧道足够）、防滥用速率限制经压测、容器自动重启/健康重部署、CI 自动构建测试部署。
- **依赖**：见 G（CI/CD）。

## E. 无障碍与交互 UX（竞品成败关键，P0–P1）

### E1. 全量 VoiceOver 走查 🔜 · P0
- **目标**：每个界面（含新角色界面/通话/导航/亲友/管理）VoiceOver 完整可用。
- **达成标准**：所有可交互元素有正确 label/hint/trait；朗读顺序合理；无"按钮、按钮"裸读；焦点管理正确；通话/求助等关键路径全程可盲操。
- **技术要点**：逐屏 VoiceOver 走查清单 + 修复；动态状态用 `accessibilityValue`/通知播报。
- **依赖**：真机 + 最好盲人用户参与。

### E2. 视觉无障碍适配 🔜（高对比大字状态条 + Dynamic Type；减少动效待补）· P1
- **目标**：覆盖**低视力**用户（不只是全盲）。
- **达成标准**：支持动态字体(Dynamic Type)放大不破版；高对比模式；减少动效尊重系统设置；色彩不依赖单一通道。
- **依赖**：设计走查。

### E3. 盲人首次上手引导 ✅（首启语音教程 + 设置可重看）· P1
- **目标**：第一次用就会用。
- **达成标准**：纯语音/触觉的 onboarding 教程教会"怎么求助、怎么避障、长按出画面、调语速"；可跳过可重看。
- **依赖**：内容设计。

### E4. 触觉精细化 ✅（按危险等级可区分震动节奏 1/2/3 下 + 回退）· P2
- **目标**：用震动模式编码方向/危险等级，减少对语音的依赖。
- **达成标准**：不同危险等级/方向有可区分的 Core Haptics 模式；可在嘈杂环境替代语音；可开关。
- **技术要点**：`CoreHaptics` 设计模式库（可核心定义参数）。

### E5. 多语言（i18n）❌ · P1
- **目标**：从仅中文到多语言（英/中起步）。
- **达成标准**：UI 与播报全量本地化；语音/标签按语言切换；竞品多语言是出海硬门槛。
- **技术要点**：`String Catalog` 本地化 + 播报文案按语言；`LabelCatalog` 多语映射。
