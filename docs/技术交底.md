## 一、系统总览与架构

### 1.1 产品定位

BeeUrEi（蜂之眼，"Be Your Eye"）是一款为视障/低视力人士打造的原生 iOS 应用，把 iPhone 的主摄像头与 LiDAR 变为"第二双眼睛"。核心能力：

- **端侧实时避障**：相机+深度数据在本机分析前方障碍，以语音/震动（声化）实时提示。
- **场景识别（看一看）**：本地识别物体、读字、识别纸币、扫码、辨色、找物、感知"周围的人"、探测光线。
- **步行导航与"我在哪"**：基于本机精确定位的路线与方位播报。
- **一键远程真人协助**：把视障用户与其选定的亲友/志愿者接通为实时语音通话；仅当用户主动按住"显示画面"时才把摄像头画面端到端传给对方。
- **聊天/群聊**：文本/语音/图片/视频/位置消息。
- **摔倒检测与紧急呼叫**：检测到疑似摔倒或用户主动触发时呼叫紧急联系人。

### 1.2 设计原则

| 原则 | 工程体现 |
| --- | --- |
| **端侧优先** | 全部视觉 AI 推理在本机完成；相机画面默认不离开设备；唯一例外是用户主动按住"显示画面"时的 P2P 直传。 |
| **隐私即安全** | 视障用户尤其脆弱：默认不发画面、知情同意、合规告知、数据最小化贯穿全栈。 |
| **无障碍优先** | 盲人侧不依赖 VoiceOver 也能听到状态——自带 TTS 总线（`SpeechHub`）；防误触二次确认；魔法轻点挂断。 |
| **可自托管 / 非商用** | 后端零原生依赖（`node:sqlite`）；代码 PolyForm Noncommercial 1.0.0，禁止向视障用户收费。 |

### 1.3 总体架构

系统由四个相互独立、边界清晰的组件构成：

```
┌─────────────────┐        WebRTC P2P 媒体(默认直连/TURN中继)        ┌─────────────────┐
│   iOS 客户端 A   │ <══════════════════════════════════════════════> │   iOS 客户端 B   │
│ (视障/盲人侧)    │                                                  │ (协助者/亲友侧)  │
└────────┬────────┘                                                  └────────┬────────┘
         │  HTTPS REST + WSS 信令                                              │
         └───────────────────────────┬──────────────────────────────────────┘
                                      ▼
                       ┌──────────────────────────────┐        ┌──────────────────────┐
                       │  自托管后端 (Fastify/TS)      │ <────> │ 管理后台 SPA (/admin) │
                       │  REST + /ws 信令 + node:sqlite│        │  纯前端 vanilla JS    │
                       └──────────────┬───────────────┘        └──────────────────────┘
                                      │
                       ┌──────────────┴───────────────┐
                       │ coturn TURN 中继 · 媒体磁盘    │
                       └──────────────────────────────┘
            官网 beeurei.hikosphere.com (独立静态站点，含法律文件 /legal/)
```

- **iOS 客户端**：Swift / SwiftUI，`@Observable @MainActor` 视图模型；WebRTC（stasel/WebRTC xcframework）；ReplayKit 录制；CoreLocation / ARKit / AVFoundation。
- **后端**：Fastify 5 + TypeScript（ESM，`tsx` 运行，`--experimental-sqlite`）；REST API + `/ws` WebSocket 信令；`Store` 抽象（内存/JSON 文件/SQLite 可替换）。装配见 `server/src/app.ts`。
- **管理后台 SPA**：`server/public/admin/`，纯前端零运行时依赖，与 API 同源，hash 路由，严格 CSP。
- **官网**：`site/`，nginx 静态托管，与 API 异源隔离；承载隐私政策/条款/EULA。

### 1.4 后端路由装配（`server/src/app.ts`）

`buildApp(store)` 在一个继承了限流钩子的子插件作用域内注册全部 HTTP 路由：认证/找回/账号/Passkey/推送/用户/亲友/拉黑/紧急/消息/群组/媒体/举报/管理/录制/**通知**/开发/导航/应用配置/求助；`/ws` 信令在独立子插件作用域注册（`registerSignaling`）。可注入项（`pushSender`/`mailer`/`appleVerifier`/`store`）便于单测与多环境。

## 二、端侧感知与视觉 AI 管线

### 2.1 采集与感知

避障管线：相机帧 + 深度（LiDAR/ARKit）→ 本机推理 → **声化（sonification）**：把"前方障碍的方位与距离"映射为空间化的声音/震动提示。整条管线在设备本地实时运行，相机帧与深度数据**不上传、不留存、不用于训练**。

**暂停丢帧的安全语义**：当会话被暂停（如切后台、相机中断、ARKit interruption）时，管线会 `pauseSession → degradeStop`，并以 `sonifier.update(nil)` 主动静音——避免在传感器数据已失效后仍按旧帧播放误导性的提示。该行为有回归测试 `HomeViewModelSafetyTests.testPausedSessionDropsInFlightFrames` 守护（断言暂停后不再产生新的非 nil 声化更新，且最后一次为静音）。

### 2.2 场景识别（看一看）

本地能力集合：物体识别、读字（OCR）、纸币识别、扫码、辨色、找物、"周围的人"、光线探测。其中**"周围的人"仅在本地估计是否有人及其大致方位，不识别任何人的身份、不存储人脸**——这是一条明确的隐私不变量，也写入隐私政策。

陈旧结果防呆、相机状态播报、暗光提示开手电、复制确认等交互细节，均围绕"盲人用户听得到、不误触、不被陈旧信息误导"展开。

### 2.3 传感器与紧急

`BeeUrEi/Sensors/EmergencyWatch.swift` 的 `EmergencyAlertCenter`（`CLLocationManagerDelegate`）实现摔倒/撞击检测与紧急呼叫：触发时**提前发起一次定位**（`requestLocation`），在发送告警时尽力附上坐标；若 `sendNow` 时定位尚未返回，则按"尽力而为"发送（不阻塞）。紧急功能在法律与产品上都被明确定性为"尽力而为"的便利功能，不替代官方急救号码。

### 2.4 盲人侧语音总线

`SpeechHub`（单例）是盲人侧的关键无障碍设施：当未开启 VoiceOver 时，`A11y.announce` 会被系统静默丢弃，导致盲人听不到通话状态/避障播报；因此盲人侧统一改走 `SpeechHub.shared.speak(..., channel:)`。语音按通道（如 `.call`）优先级让位/被让位，避免避障、导航、识别、通话状态互相打断。

## 三、实时通话与 WebRTC 信令

### 3.1 媒体引擎（`BeeUrEi/RemoteAssist/MediaEngine.swift`）

`MediaEngine` 协议抽象"真实音视频"，注入式以便单测；具体实现 `WebRTCMediaEngine`（需 stasel/WebRTC xcframework），无包时回退 `StubMediaEngine`（**不伪装连接**，`start` 即如实上报媒体失败，杜绝"假装已接通"）。

视频隐私模型：
- 协助者（`asCaller=false`）：不发视频、收对方视频 + 双向语音。
- 视障侧（`asCaller=true`）：默认只加一条 `isEnabled=false` 的视频轨、**且不开相机采集**；仅当 `setLocalVideoSending(true)` 时才 `startCapture` 并启高画质（最高 ~2.5 Mbps / 720p，`degradationPreference=balanced` 拥塞自动降级）。停发即 `stopCapture`，确保相机只在用户主动发画面期间运转、隐私指示灯与"未发送"一致。

关键工程细节：远端候选缓存（`pendingCandidates`，须先 `setRemoteDescription` 成功再 `add` 候选，否则被静默丢弃）、`onMediaStateChange` 把"信令已连接但媒体没通"暴露给 UI、`pollStats` 每 2s 以活跃候选对 RTT 映射"信号强弱"。音频会话经 `RTCAudioSession` 配为 `.playAndRecord/.voiceChat`、默认走扬声器（盲人免持）。

### 3.2 信令房间（`server/src/signaling/hub.ts`）

`SignalingHub` 是纯逻辑、可单测的房间登记表：`join/leave/peersInCall/isOnline/activeCalls`。一个 `callId` 即一个房间。`Member` 含 `clientId/userId/role/callId/joinedAt?/caps?`。`activeCalls()` 按 callId 聚合参与者、取最早加入为开始时间、标记是否有管理员旁观，供管理后台实时总览。

### 3.3 `/ws` 信令协议（`server/src/routes/ws.ts`）

客户端连接 `/ws?token=<JWT>`，首帧发 `{type:'join', callId, role, caps}`，随后中继 `offer/answer/ice/video-gate/end/control/record-*/obs-*`。

**参与权校验（防窃听/劫持）**：`join` 时校验该 `callId` 的真实参与者——来源为定向亲友呼叫登记（`pendingCalls`，180s TTL）或公开求助队列（`openHelp`，认领后含志愿者），两表跨表去重防影子覆盖；非参与者一律 `close(4003)`。1:1 房间最多两名**参与者**（旁观管理员不计入）。

呼叫建立时序（亲友呼叫）：

```
盲人A                          后端(/ws + pendingCalls)                协助者B
  │  POST /api/assist/call (callId, targetUserIds) ──────────────────────► 登记 pendingCalls
  │  (APNs/VoIP 唤起 B 的 CallKit 来电) ─────────────────────────────────► [来电界面]
  │  WS join(callId, role=blind, caps) ──►校验参与权──► joined           │
  │                                       ◄── WS join(callId, role=helper)│
  │  ◄── peer-joined ───────────────────────────────► peer-joined ──────►│
  │  createOffer → offer ──► 中继(过滤非本通话/旁观) ──► offer ──────────►│
  │  ◄──────────────────────── answer ◄─── 中继 ◄────────── answer ──────│
  │  ◄═══════════════ ICE/媒体直连(或 TURN 中继) ═══════════════════════►│
```

### 3.4 ICE/TURN

`GET /api/assist/turn`（鉴权）下发 STUN + **短时效 TURN 凭据**（基于 `TURN_SECRET` 的时限用户名/密码，TTL 6h）。无法 P2P 直连时，媒体经自托管 coturn 在传输途中转发——**仅转发，不存储、不查看**。

## 四、管理员合规旁观网格

### 4.1 设计目标与合规边界

为保障安全、保护脆弱用户并防止滥用，管理员可对一通进行中的通话进行**合规旁观**：观看/收听音视频、在通话中开麦说话、必要时强制结束。**核心合规不变量：绝非隐蔽——开始旁观时实时通知通话双方**（不可关闭的告知横幅 + 盲人侧语音播报）。该设计在产品上拒绝了"不通知用户"的隐蔽监听方案。

### 4.2 服务端准入（`server/src/routes/ws.ts`）

管理员发 `{role:'admin', observe:true, caps:['adminObserver']}`，服务端门控：

1. **DB 角色校验**：`store.findById(auth.sub).role === 'admin'`（不信任客户端自报），否则 `close(4003,'not_admin')`。
2. 房间须有进行中通话（`peersInCall` 非空），否则 `call_not_active`。
3. 同房间至多一名旁观者，否则 `observer_exists`。
4. **能力门控**：`existing.every(p => p.caps.includes('adminObserver'))`——**所有**现有参与者都声明支持旁观告知才允许，否则 `call_not_observable`。这保护旧版客户端：无法显示告知横幅的客户端不会被旁观。
5. 准入后向双方发 `peer-joined {role:'admin', ...}`，触发其客户端显示告知横幅 + 语音。

并发安全加固（对抗式复审定稿）：普通 `join` 分支**净化客户端自报角色**（仅接受 `blind/helper`），杜绝参与者自报 `role:'admin'` 伪造"管理员监看"告知或污染实时总览。

### 4.3 obs-* 定向中继与媒体隔离

旁观媒体走独立的 `obs-offer/obs-answer/obs-ice` 信令，**带 `to` 字段定向**到具体参与者；中继规则：

```js
let targets = peersInCall(callId, sender)
if (msg.to) targets = targets.filter(p => p.userId === msg.to)        // obs-* 定向
else if (msg.type in {offer,answer,ice}) targets = targets.filter(p => p.role !== 'admin')  // 1:1 主媒体不泄漏给旁观管理员
```

即：1:1 主媒体（盲人↔协助者）**绝不**发给旁观管理员；管理员的音视频走独立 obs 通道。

### 4.4 iOS 旁观 PC 网格（`MediaEngine.swift`）

管理员侧为每个被旁观参与者建一条**与主 pc 隔离**的 `observerPCs[peerId]`，共享同一本地音视频轨；`ObserverPCDelegate` 须强引用（RTCPeerConnection 仅弱引用 delegate）。参与者侧把自己的音视频轨加到对管理员的 obs PC，**按 `isEnabled` 门控**——所以"管理员见即所见"：用户没分享画面，管理员就看不到画面（只有音频）。

旁观握手方向：既有参与者向新加入的管理员发 offer（`addObserverPeer(adminId, offer:true)`），管理员应答（`offer:false`）。离场（`stopObserving`/界面消失）**不向参与者发 `end`**（不结束他人通话）；强制结束是单独动作，走鉴权 REST `POST /api/admin/calls/:id/end`（可靠，不依赖本端 WS 帧时序）。

### 4.5 浏览器内 Web 旁观（`server/public/admin/app.js` · `startObserver`）

管理后台 SPA 内置一个 vanilla-JS WebRTC 旁观客户端：取 ICE（`/api/assist/turn`）→ 开 `wss://…/ws?token=` → 发 observe join → 处理 `joined/peer-joined/obs-*/peer-left/end` → 为每个参与者建 `RTCPeerConnection` 应答其 obs-offer、`ontrack` 渲染到视频块 → 可开麦说话（`getUserMedia` 音频 + 重协商）、强制结束、结束旁观。被服务端拒绝（`call_not_observable` 等）时优雅提示并收起。

## 五、通话录制：采集、同意、留存与回看

### 5.1 采集（`BeeUrEi/RemoteAssist/CallRecorder.swift`）

基于 ReplayKit（`RPScreenRecorder`）采集通话画面（含双方音视频呈现），输出 `.mov`。`isAvailable` 仅真机为真。

### 5.2 服务端权威知情同意（`server/src/recording/consentRegistry.ts`）

**同意不可由发起者伪造**：被录方须经鉴权端点 `POST /api/recordings/consent` 亲自授予；服务端 `RecordingConsentRegistry` 登记（callId→granter→expiry，6h TTL）。授予者须为该通话的**真实参与者**（与信令同一参与权校验：`pendingCalls`/`openHelp`/持久 `CallRecord` 兜底）——杜绝串通的第三方为不在的通话伪造同意。`POST /api/recordings` 创建录制时，`consentBy` 由服务端从同意登记表取（`consenters()` 排除发起者），策略要求同意时至少一名有效同意者。

### 5.3 元数据（"时间地点人 + 时长"）

`Recording`（`server/src/db/store.ts`）字段：`id/callId/ownerId/consentBy[]/reason/recordedAt/mediaId?` + `participants?`（发起者+同意者，去重持久化）+ `durationSec?` + `lat?/lon?/locationLabel?` + `deletedAt?`。位置由客户端在 `finishAndUpload` 时**仅在定位已授权时**采集（`bestEffortLocation`，绝不为录制弹新授权框）；时长由录制起止差计算。

### 5.4 软删除 / 合规留存

- **用户软删除** `DELETE /api/recordings/mine/:id`：仅 owner 可调，**只置 `deletedAt`**（对用户隐藏），**不删媒体文件**。
- `recordingsForUser` 过滤 `deletedAt`（用户看不到）；`allRecordings` 保留（管理员仍可见，标注"用户已删除·留存中"）。
- **物理清除**统一由 `sweepExpiredRecordings`（`server/src/recording/retention.ts`）按 `recordedAt + retentionDays`（默认 7 天）执行；软删除**不重置**该时钟。
- **取证留存**：被未结举报引用为证据的录制即使过期也不清除（`reportsCitingRecording(id).some(status !== 'resolved')`），直到举报处置完毕。

### 5.5 回看流式端点（`server/src/routes/recordings.ts`）

`GET /api/recordings/:id/media`：HTTP **Range** 流式（206 局部 / 200 全量，416 不可满足），便于拖动。**双路鉴权**：① `Authorization: Bearer`（App，且回库核验 status active + tokenVersion 匹配）；② 短时签名**媒体令牌** `?t=`（Web `<video>` 不能带头）。授权 = owner（且未软删除）∨ admin（admin 可看软删除项，且**查看入审计** `recording.view`）。

**安全加固（复审）**：录制媒体**严禁经通用 `/api/media/:id` 外泄**——通用端点对 `store.recordingByMediaId(id)` 命中者一律返回 404，强制走录制作用域端点（杜绝"owner 的好友经好友授权下载录制"）。媒体令牌绑定 `recordingId + 用户 + tokenVersion`，60s 过期（单一真源 `MEDIA_TOKEN_TTL_SEC`），改密/封禁/强制下线即失效。

### 5.6 客户端回看 UI

- 用户端"我的录音"（`BeeUrEi/Features/Account/MyRecordingsView.swift`）：列出本人录制（时间/地点/人/时长），下载到本地 `.mov` 后用 `AVPlayer`（复用 `VideoPlayerSheet`）回放；可软删除（删后清本机缓存副本）。
- 管理端（`AdminHomeView`）：全站录制总览（含已软删除）+ 回放 + 彻底删除（被未结举报引用者 409 拒绝 + 删除入审计）。
- 管理 Web：`<video>` + 短时令牌播放。

## 六、后端架构与数据模型

### 6.1 Store 抽象与可替换实现

`Store` 接口（`server/src/db/store.ts`）把上层与持久化解耦：
- `MemoryStore`（测试）→ `JsonFileStore extends MemoryStore`（零依赖文件持久化）→ `SqliteStore`（生产，`node:sqlite`）。
- 任何数据模型变更须**同时**改：接口 + MemoryStore + JsonFileStore 的载入/保存 + SqliteStore 的 DDL/迁移/INSERT/行映射——漏一处即静默丢字段。

### 6.2 加性迁移（`server/src/db/sqliteStore.ts`）

`CREATE TABLE IF NOT EXISTS` + 每个后加列用 `try { db.exec('ALTER TABLE x ADD COLUMN y T') } catch {}`——旧库就地升级、零迁移脚本、完全向后兼容。数组/对象列以 JSON TEXT 存、用容错的 `parseJsonOr` 解析（一行坏数据不致整请求 500）。

## 七、认证、授权与安全模型

### 7.1 令牌与会话（`server/src/auth/tokens.ts`）

- **JWT access**（1h）携带 `sub/role/tv`；**不透明 refresh**（30d，仅哈希入库）。
- **`tokenVersion` 即时吊销**：改密、封禁、强制下线即递增，使该用户已签发的 access token 全部失效（`requireAuth` 回库比对 `tv`）。
- **JWT_SECRET fail-closed**：缺失或过短即拒绝启动，绝不使用硬编码兜底密钥。

### 7.2 授权（`server/src/auth/rbac.ts`）

`requireAuth([roles])` preHandler：验签 → 回库核验账号存在、status active、tokenVersion 匹配 → 角色门控。`requireFeature(store, feature)` 叠加全站/单用户功能门控。

### 7.3 其它安全机制

- **Passkey/WebAuthn**：抗钓鱼登录，私钥不离设备安全区。
- **bcrypt** 加盐哈希密码。
- **短时媒体令牌**：见 §5.5，绑定 recordingId+用户+tokenVersion，60s。
- **速率限制**：全局 + 敏感端点（发码 60s 冷却 + 窗口上限、媒体上传 10/min）。
- **CSP**：管理后台 `default-src/style-src 'self'`，无内联样式（动态尺寸走 CSSOM）；录制媒体经同源 `<video>` 播放，CSP 无需放宽。
- **信令防窃听**：参与权校验 + 房间容量 + obs-* 隔离（§3.3 / §4）。
- **录制媒体不外泄**：通用 `/api/media` 拒绝录制媒体（§5.5）。
- **审计**：所有后台变更与管理员查看录制入 `admin_audit`。

### 7.4 安全不变量清单（节选）

1. 客户端自报角色绝不授予特权（admin 只能经 DB 校验路径获得）。
2. 录制同意只能由真实被录参与者亲授，服务端权威。
3. 1:1 主媒体绝不发给旁观管理员；旁观必通知双方。
4. 录制媒体只能经 owner∨admin 的录制端点访问；软删除对用户隐藏、对 admin 在留存期内保留。
5. 举报证据仅录制拥有者可附；被未结举报引用的录制不被清理。
6. 举报处理结果通知中，举报人不获知对对方的具体处罚（per-recipient 数据）。

### 7.5 实名认证（KYC，人工审核 · `server/src/kyc/*` · `routes/kyc.ts` · `routes/admin.ts`）

诚实约束：真正的自动证件核验需第三方持牌服务，故本系统做成**真人后台审核**——用户提交实名 + 证件 + 自拍 → 管理员对照人工审核 → 通过/拒绝；**绝无 mock 自动通过**。

- **信封加密（`kyc/crypto.ts`）**：每条记录一把随机 256-bit DEK，正文（姓名/证件号/图片）以 AES-256-GCM 加密；DEK 再由独立主密钥 `KYC_ENC_KEY` 包裹（信封）。正文与包裹两层 GCM 都绑定 AAD=`submissionId|kind|keyId`，密文/密钥无法被挪用到别的记录或字段。`KYC_ENC_KEY` 与 `JWT_SECRET` **必须不同**且 fail-closed（缺失/弱/相同即拒启动）——令牌密钥泄露不能顺带解密任何证件。
- **隔离落盘（`kyc/storage.ts`）**：证件密文存独立目录 `KYC_DIR`（0700，UUID 文件名），**绝不进 `media` 表、绝不经 `/api/media`**；仅 admin 审核端点解密读取。
- **图片归一化（`kyc/imageNormalize.ts`）**：magic-byte 嗅探 + 剥离 JPEG(APP1/APP13/COM)/PNG 文本块（去 GPS/EXIF）+ 结构校验拒绝畸形/多格式文件；客户端另在上传前重编码为 JPEG（纵深防御）。
- **数据最小化与无 PII 外泄**：`publicUser`/`selfView` 仅含 `verified` 布尔徽章；姓名/证件号/图片**只**在唯一的 admin 审核详情/证件图端点解密，且每次访问（`kyc.view`/`kyc.view-doc`）必入审计。用户提交后无任何端点可取回自己的证件原图或回读姓名。
- **留存（`kyc/retention.ts`）**：拒绝即清证件图 + 姓名 + 证件号；通过保留加密姓名（徽章法律依据）、7 天宽限后清证件图与证件号；停滞 pending 30 天自动拒 + 清；级联删号清证件（法务保留豁免）。每小时清扫。
- **决策恰好一次**：`decideVerification` 为条件更新（`WHERE status='pending'`），两管理员竞态仅一人生效副作用；一人一活跃记录由应用守卫 + sqlite 部分唯一索引双保险。
- **职责分离**：管理员不得审核/查看/撤销/保留**本人**提交（detail/doc/approve/reject/revoke/hold 全部 `v.userId === adminId → 403`）。
- **安全不变量（新增）**：7. KYC 解密仅经 `adminOnly` 路由，证件密文 DB 单独泄露或磁盘单独泄露均不可解（需磁盘 + 库 + `KYC_ENC_KEY` 三者）。8. 徽章服务端权威（`identityVerified`），客户端不可伪造。

## 八、通知、举报与内容治理

### 8.1 举报与处置（`server/src/routes/{reports,admin}.ts`）

举报 `POST /api/reports`（可附 `evidenceRecordingId`，**仅拥有者可附**，去重时不吞掉不同的新证据）。管理员处置阶梯 `POST /api/admin/reports/:id/moderate`：`dismiss/warn/suspend/ban` → 落处置结果 + 对被举报人施加后果（警告/封禁+撤销会话）+ 审计；含"不能处置自己/最后一名管理员"保护。

### 8.2 处置通知双方（`server/src/notifications/notify.ts`）

**持久化收件箱 + 离线推送**：`notifyUser` 先写 `notifications` 表（权威、登录后必见），再 best-effort `sendAlert`（APNs，未配置/离线则 Noop）。处置时通知举报人与被举报人。**隐私设计**：每收件人单独构造结构化数据——举报人的通知**不含 `decision`**（不泄漏对对方的具体处罚），被举报人可知关于自己的结果；文案双语、按收件人语言。通知写入用 try/catch 包裹，绝不让通知失败 500 掉已提交的处置。`GET /api/notifications` 严格作用域到 `req.user.sub`。

### 8.3 其它治理

拉黑、内容过滤（敏感词）、全站功能开关/公告/维护模式、管理员实时通话总览与强制结束、管理后台 SPA 概览/用户/关系/通话/录制/举报/审计视图，均见 `server/src/routes/admin.ts` 与 `server/public/admin/`。

## 九、隐私与合规工程

### 9.1 端侧优先与数据最小化

视觉推理全本地；相机帧默认不离设备、不留存、不训练；唯一例外是用户主动按住"显示画面"的 P2P 直传。"周围的人"不识别身份不存人脸。位置：导航/我在哪本地用精确定位但不上传精确坐标；求助/紧急只附粗略（省/市/区）位置；录制位置仅在已授权时采集。

### 9.2 知情同意与告知

录制需双方同意（服务端权威）；管理员旁观必实时告知双方；诊断数据（Sentry）须显式开启。撤回同意不影响撤回前合法处理。

### 9.3 留存与删除

自动清理（录制默认 7 天、诊断 ≤90 天、验证码短时、备份 ≤35 天滚动覆盖）；软删除/合规留存（§5.4）；账号删除级联（§6.3）。

### 9.4 法律映射

每类数据在隐私政策中载明 GDPR 第 6 条合法性基础；数据主体权利（访问/更正/删除/限制/可携/反对/撤回）+ CCPA/CPRA 权利；跨境传输（基础设施在日本，受欧盟充分性认定覆盖；其它接收方依 SCC/减损）；特殊类别数据（健康/残障）依明示同意/重大利益。第三方处理者：Apple（登录/推送/地图/CallKit/分发）、高德 AMap（中国大陆导航，仅必要坐标）、Open-Meteo（仅降精度坐标）、Sentry（诊断）、NetEase 163（SMTP）、AWS（托管）。法律文件 v3.0 与本系统当前能力（录制/旁观/证据/通知）逐项对应，见 `site/public/legal/` 与 `BeeUrEi/Consent/LegalText.swift`。

## 十、部署、运维与质量保障

### 10.1 部署拓扑

- 后端容器 `beeurei-api`（Docker，`node:sqlite` + 媒体磁盘，数据卷 `beeurei-data`）运行于 AWS 东京 `ap-northeast-1`。
- coturn `beeurei-coturn`（host 网络，UDP/TCP 3478 + UDP 49160-49200）。
- 官网/管理后台静态资源容器化（nginx）。
- 对外经 **Cloudflare Tunnel**（token 托管）暴露 `beeurei-api.hikosphere.com`、`beeurei.hikosphere.com`；TLS 终止于 Cloudflare。
- 管理后台 `/admin/`（与 API 同源）；法律文件迁至官网同源 `/legal/`。

### 10.2 环境与密钥

`JWT_SECRET`（fail-closed）、`TURN_SECRET`（与 coturn 一致）、APNs `.p8`（VoIP 与 alert 共用，topic 区分 `.voip`）、SMTP（163，端口 994）、`SENTRY_DSN`、`METRICS_TOKEN`、`ADMIN_USERNAME/PASSWORD`（引导管理员）、`DB_DRIVER/DB_PATH/MEDIA_DIR`。改 `.env` 后须 `stop/rm/run` 重建容器（restart 不重读 env-file）。

### 10.3 推送链路

来电经 APNs **VoIP**（PushKit）唤起 CallKit 系统来电界面 + 普通 **alert** 兜底；提醒类（好友请求/举报结果等）经 alert。

### 10.4 可观测性

`/metrics`（Prometheus 文本，可选 `METRICS_TOKEN` Bearer）、`/health` 与 `/api/ready`（触达存储）、Sentry 错误监控。后台定时器每小时跑录制留存清理（不依赖管理员访问）。

### 10.5 测试与质量保障

- **服务端**：`vitest`，约 **289** 项（含信令参与权/旁观准入/录制同意/回看 Range+授权/软删除+合规留存/举报证据/通知/迁移等安全回归）。
